# AGENTS.md

Agent orientation for the `al2023-devops` repository.

## What this repo is

A single-`Dockerfile` project that builds a multi-arch (amd64 + arm64) Docker image based on Amazon Linux 2023, bundling 80+ DevOps tools for CI/CD pipelines and EKS workload administration. Published to Docker Hub (`erhhung/al2023-devops`) and GHCR.

No package manager, no test suite, no lockfiles. The entire build is shell scripts mounted into Docker via `--mount=type=bind`.

## Local build

```bash
./build.sh                          # build for host arch, load into local Docker
./build.sh --platform linux/amd64   # cross-build for amd64
./build.sh --platform linux/arm64   # cross-build for arm64
```

Prerequisites:
- A BuildX builder named `multi-builder` must exist: `docker buildx create --name multi-builder --bootstrap --use`
- AWS CLI profile `github` must be configured — `build.sh` logs in to ECR Public (`public.ecr.aws`) using it before the build
- Build output is appended to `build.log` (gitignored)

There is no `Makefile`, `Taskfile`, or `justfile` in this repo. `build.sh` is the only build entry point.

## CI/CD pipeline

Triggered **only by Git tags** matching `[2-9][0-9].[0-1][0-9].?[0-3]?[0-9]?` (e.g. `26.07`, `25.12.31`). Push to branches does NOT trigger CI.

Tag format convention: `YY.MM` or `YY.MM.DD`. Tags should be GPG-signed (`git tag -s`).

CI flow:
1. `launch-runner` — provisions an EC2 ARM64 spot instance as a self-hosted runner via AWS OIDC
2. `build-amd64` (ubuntu-latest) and `build-arm64` (EC2 self-hosted) run in parallel, each pushing a per-platform digest
3. `merge-manifests` combines the digests into a multi-arch manifest and pushes to both registries
4. `terminate-runner` and `notify-slack` always run at the end

## Dockerfile structure

Three stages — agents editing the Dockerfile must understand the boundaries:

| Stage | Base | Purpose |
|---|---|---|
| `builder` | `amazonlinux:2023` | Compile tools from source (Python, Buildah, Skopeo, etc.) |
| `consolidator` | `scratch` | Collect `/usr/local/` artifacts + dotfiles into a clean layer |
| `final` | `amazonlinux:2023` | Install binary/package tools via shell scripts; sets all `ENV` vars |

Scripts are never `COPY`'d into the image — they are mounted with `--mount=type=bind,source=scripts/...,target=/tmp/install.sh` and run once. `/tmp` is also mounted as `tmpfs` in the `final` stage to avoid leaving temp files in layers.

## Directory map

```
scripts/build/       # Source-compiled tools (run in builder stage)
scripts/install/     # Binary/package installs (run in final stage)
scripts/usrlocalbin/ # Custom scripts deployed to /usr/local/bin inside image
scripts/utils/       # Developer utilities (NOT used in Docker build)
scripts/versions.sh  # Last RUN in final stage; generates /root/.versions.json
config/              # Dotfiles COPY'd into /root/ (via consolidator stage)
.github/workflows/   # CI: ci.yml (main), build.yml (per-platform), merge.yml (manifest)
```

## Shell script conventions

All scripts use `set -euxo pipefail`. Any change to a script will bust the Docker layer cache for that `RUN` step and all subsequent steps.

Version discovery pattern used throughout (fetches latest GitHub release tag):
```bash
curl -ILs "$RELEASES_URL/latest" | sed -En 's/^location:.+\/tag\/v(.+)\r$/\1/p'
```

Architecture detection pattern:
```bash
ARCH=$(uname -m | sed -e 's/aarch64/arm64/' -e 's/x86_64/amd64/')
```

GitHub Actions log grouping used in all scripts:
```bash
echo "::group::Step description"
trap 'echo "::endgroup::"' EXIT
```

## Key env vars set in the image (Dockerfile `ENV`)

These are baked into the image and affect tool behavior; know them before modifying scripts:

- `PATH` is extended incrementally across multiple `ENV` lines: `/root/.local/bin`, `/usr/local/poetry/bin`, `$PNPM_HOME/bin`, `$JAVA_HOME/bin`, `/usr/local/maven/bin`, `/root/.krew/bin`
- `JAVA_HOME=/usr/lib/jvm/java-26-amazon-corretto`
- `PNPM_HOME=$XDG_DATA_HOME/pnpm` (`$XDG_DATA_HOME=/root/.local/share`)
- `HELM_BIN=/usr/local/bin/helm` — required by the `helm-git` plugin
- `BUILDAH_ISOLATION=chroot` — needed for rootless builds inside containers
- `TF_CLI_ARGS_init/plan/apply=-compact-warnings`

## Notable files inside the image

- `/root/.versions.json` — generated by `scripts/versions.sh`; manifest of all installed tool versions
- `/usr/local/bin/git-credential-store-ro` — read-only Git credential helper (passes `get`, ignores `store`/`erase`)
- `/usr/local/bin/kustomize.sh` — wrapper that deduplicates YAML keys with `yq` before running `kustomize build`

## What does NOT exist here

- No tests — there is no way to run a test suite locally
- No linter config (`.shellcheckrc`, etc.) — shellcheck directives are inline comments
- No pre-commit hooks
- No `.env.example` — secrets are GitHub Actions repository variables/secrets only
- `scripts/utils/dhrmi.sh` and `ghrmi.sh` are dev utilities for cleaning old registry images; they are not part of the build
